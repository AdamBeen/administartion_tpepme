# CONTEXTE COMPLET DU PROJET — ADMINTPE GRAPHRAG V1
# Copilote interne d'instruction administrative TPE/PME
# Stack cible : LangGraph + Cassandra + Obsidian + Frontend Web
# Version : V1 volontairement non sécurisée

============================================================
1. PARAGRAPHE CONTEXTUEL DU USE CASE
============================================================

Le projet consiste à créer un copilote interne d'instruction administrative destiné à des agents administratifs chargés de traiter des dossiers TPE/PME au Maroc. Le système ne s'adresse pas aux citoyens lambda : il est utilisé uniquement par un administrateur ou un agent instructeur qui pose une question métier à propos d'un dossier client/société fourni au moment de l'analyse. Le cas d'usage couvre trois familles principales : les demandes d'aides publiques ou économiques, les autorisations administratives liées à l'activité d'une TPE/PME, et les recours administratifs après refus, incomplétude ou contestation. L'objectif est d'aider l'agent à lire rapidement un dossier complexe, repérer les pièces présentes et manquantes, identifier les incohérences, retrouver les règles administratives pertinentes, exploiter un graphe de connaissances des démarches et produire une réponse JSON structurée avec recommandation, niveau de confiance et déclenchement HITL métier. La V1 doit volontairement négliger la sécurité afin de servir de baseline fonctionnelle vulnérable qui sera ensuite documentée, red-teamée et sécurisée dans une V2.

============================================================
2. OBJECTIF GLOBAL DU PROJET
============================================================

Construire un workflow complet d'analyse administrative hybride combinant :

- LangGraph pour orchestrer les agents et les étapes du workflow.
- Cassandra pour stocker le Knowledge Graph administratif, les embeddings, les résultats d'analyse, les logs et les traces.
- Obsidian pour visualiser le Knowledge Graph sous forme de fichiers Markdown interconnectés.
- Un frontend web pour permettre à l'administrateur de :
  - poser une question sur un dossier client/société ;
  - uploader un dossier client contenant PDF, Excel, CSV, XLS, XLSX ou fichiers d'emails ;
  - déclencher l'analyse ;
  - consulter une réponse nettoyée et organisée ;
  - voir les pièces manquantes en gras ;
  - consulter un panneau d'error logging avancé permettant de comprendre les erreurs du workflow.

La V1 ne doit PAS intégrer de sécurité active. Il faut volontairement négliger les protections contre prompt injection, documents malveillants, exfiltration, contrôle d'accès avancé, anonymisation automatique ou filtrage de contenu. Ces limites devront être documentées comme vulnérabilités connues.

============================================================
3. CLARIFICATION FONDAMENTALE SUR LES DONNÉES
============================================================

Le dossier société/client n'est PAS stocké dans le GraphRAG.

La séparation doit être stricte :

1. RAG vectoriel :
   - Contient un seul document principal décrivant le contexte général de l'administration marocaine pour les TPE/PME.
   - Ce document est chunké, indexé et utilisé pour récupérer des passages textuels pertinents.

2. Knowledge Graph / GraphRAG :
   - Contient uniquement la connaissance administrative générale et réutilisable.
   - Il ne contient pas les dossiers réels des sociétés.
   - Il contient les relations entre démarches, aides, autorisations, recours, conditions, pièces, services, décisions, motifs, délais et niveaux HITL.

3. Dossier client/société :
   - Fourni uniquement en entrée runtime via upload frontend.
   - Analysé temporairement pendant l'exécution du workflow.
   - Transformé en résumé structuré temporaire.
   - Utilisé pour interroger le RAG vectoriel et le GraphRAG.
   - Ne doit pas être injecté comme connaissance persistante dans le Knowledge Graph V1.

Phrase à respecter dans la documentation :

"Le GraphRAG ne stocke pas les dossiers des sociétés. Il stocke uniquement la connaissance administrative structurée. Les dossiers sociétés sont fournis par l'administrateur au moment de la question et sont traités comme contexte temporaire non persistant dans le graphe métier."

Le périmètre V1 couvre uniquement :

A. Demandes d'aides TPE/PME

C. Recours administratifs
- recours après refus ;
- recours après dossier incomplet ;
- recours contre une décision administrative ;
- demande de réexamen ;

Hors périmètre V1 :
- diagnostic juridique définitif ;
- décision administrative automatique ;
- communication directe avec le citoyen ;
- signature ou validation officielle d'une décision ;
- traitement médical, fiscal complexe ou bancaire réel ;
- connexion à des systèmes gouvernementaux réels ;
- données réelles de citoyens ou de sociétés en production.

11. LANGGRAPH — NŒUDS À IMPLÉMENTER
============================================================

Le workflow LangGraph doit être explicite, traçable et debuggable.

État global recommandé : AdminTPEState

Champs de l'état :
- run_id
- question_administrateur
- uploaded_files
- extracted_texts
- dossier_temporaire
- classification_demande
- vector_query
- vector_results
- graph_query
- graph_paths
- merged_context
- raw_llm_output
- parsed_json
- hitl_result
- cleaned_response
- errors
- debug_trace
- timings

Nœuds LangGraph :

1. node_receive_input
Rôle : recevoir question + fichiers uploadés.
Entrée : question, fichiers.
Sortie : état initial avec run_id.

2. node_validate_input_basic
Rôle : validation technique minimale, pas sécurité.
Vérifie uniquement : présence d'une question, présence possible de fichiers, formats supportés.
Ne bloque pas les contenus suspects.

3. node_extract_files
Rôle : extraire le texte des PDF, CSV, XLS, XLSX et fichiers d'emails.
Sortie : textes bruts par fichier.
Doit logger toutes les erreurs d'extraction par fichier.

4. node_build_temporary_dossier_summary
Rôle : utiliser un LLM simple pour extraire les faits du dossier fourni en entrée.
Sortie : dossier_temporaire JSON.
Important : ne pas persister ce dossier dans le Knowledge Graph.

5. node_classify_admin_request
Rôle : classifier la demande : aide, autorisation, recours, mixte, inconnu.
Sortie : classification_demande.

6. node_build_vector_query
Rôle : transformer question + dossier_temporaire en requête pour le RAG vectoriel.
Sortie : vector_query.

7. node_vector_retrieve
Rôle : interroger les chunks du document administratif général.
Sortie : vector_results.

8. node_build_graph_query
Rôle : transformer question + classification + dossier_temporaire en requête GraphRAG.
Sortie : graph_query.

9. node_graph_retrieve_paths
Rôle : rechercher dans Cassandra les chemins relationnels pertinents.
Sortie : graph_paths.

10. node_merge_contexts
Rôle : fusionner question, résumé dossier, RAG vectoriel et GraphRAG.
Sortie : merged_context.

11. node_administrative_analysis_llm
Rôle : produire l'analyse administrative en JSON strict via LLM simple.
Sortie : raw_llm_output.

12. node_parse_and_validate_json
Rôle : vérifier que la sortie est un JSON valide.
Si JSON invalide : tenter une réparation simple via LLM ou fonction de parsing.
Sortie : parsed_json ou erreur.

13. node_hitl_router
Rôle : appliquer les règles métier HITL V1.
Sortie : hitl_result.

14. node_clean_response_for_frontend
Rôle : transformer le JSON en réponse organisée pour affichage frontend.
Sortie : cleaned_response.

15. node_persist_run_logs
Rôle : stocker logs, erreurs, timings, résultat JSON et trace dans Cassandra.
Sortie : confirmation persistence.

16. node_error_handler
Rôle : centraliser les erreurs et produire une sortie frontend compréhensible.
Sortie : erreur structurée.

14. HITL MÉTIER V1
============================================================

Le HITL de V1 n'est pas une couche de sécurité. C'est uniquement une logique métier d'escalade.

Niveaux :

VERT :
- dossier relativement simple ;
- pas d'incohérence majeure ;
- pièces principales présentes ;
- recommandation exploitable par l'agent.

ORANGE :
- pièce importante manquante ;
- condition non vérifiable ;
- incohérence documentaire ;
- recours administratif ;
- faible niveau de confiance ;
- demande de complément nécessaire.

ROUGE :
- suspicion de fraude ;
- conflit fort entre documents ;
- décision de rejet potentielle ;
- recours sensible ;
- dossier très incomplet ;
- risque d'impact juridique ou financier élevé.

Règles simples :

Si pieces_manquantes non vide -> au minimum ORANGE.
Si incoherences_detectees non vide -> au minimum ORANGE.
Si type_demande = recours -> au minimum ORANGE.
Si suspicion_fraude dans risques -> ROUGE.
Si decision_proposee = rejet_potentiel -> ROUGE.
Si niveau_confiance < 0.65 -> ORANGE.
Si niveau_confiance < 0.45 -> ROUGE.

15. CASSANDRA — TABLES RECOMMANDÉES
============================================================

Keyspace : admintpe_v1

Table 1 : kg_nodes
- node_id text PRIMARY KEY
- node_type text
- label text
- description text
- properties map<text,text>
- tags set<text>
- created_at timestamp
- updated_at timestamp

Table 2 : kg_edges
- edge_id text PRIMARY KEY
- source_node_id text
- target_node_id text
- relation_type text
- description text
- weight double
- properties map<text,text>
- created_at timestamp

Créer des index secondaires ou tables de requête dédiées selon les besoins :

Table 3 : kg_edges_by_source
- source_node_id text
- relation_type text
- target_node_id text
- edge_id text
- PRIMARY KEY ((source_node_id), relation_type, target_node_id)

Table 4 : kg_edges_by_target
- target_node_id text
- relation_type text
- source_node_id text
- edge_id text
- PRIMARY KEY ((target_node_id), relation_type, source_node_id)

Table 5 : vector_chunks
- chunk_id text PRIMARY KEY
- document_id text
- source_name text
- section text
- chunk_text text
- embedding vector<float, dimension> si supporté par Cassandra/Astra
- metadata map<text,text>
- created_at timestamp

Si la version Cassandra utilisée ne supporte pas vector nativement :
- stocker embeddings dans Astra Vector ou autre vector store ;
- stocker uniquement chunk_id, metadata et texte dans Cassandra.

Table 6 : workflow_runs
- run_id text PRIMARY KEY
- question text
- status text
- type_demande text
- created_at timestamp
- completed_at timestamp
- total_duration_ms int
- final_json text
- cleaned_response text

Table 7 : workflow_errors
- run_id text
- timestamp timestamp
- node_name text
- error_type text
- error_message text
- stack_trace text
- user_explanation text
- suggestion text
- PRIMARY KEY ((run_id), timestamp, node_name)

Table 8 : workflow_traces
- run_id text
- step_index int
- node_name text
- input_summary text
- output_summary text
- duration_ms int
- status text
- PRIMARY KEY ((run_id), step_index)

Table 9 : graphrag_paths
- run_id text
- path_id text
- path_text text
- nodes list<text>
- relations list<text>
- interpretation text
- PRIMARY KEY ((run_id), path_id)

- Message d'erreur brut
- Explication lisible
- Suggestion de correction
- run_id

9. Mode debug

============================================================
18. ERROR LOGGING AVANCÉ
============================================================

Le système doit logguer chaque étape.

Format d'un log :

{
  "run_id": "",
  "timestamp": "",
  "node_name": "node_extract_files",
  "status": "error|warning|success",
  "duration_ms": 0,
  "input_summary": "",
  "output_summary": "",
  "error": {
    "type": "",
    "message": "",
    "stack_trace": "",
    "user_explanation": "",
    "suggestion": ""
  }
}

Exemples d'explications lisibles :

Erreur extraction PDF :
"Le fichier PDF n'a pas pu être lu. Il peut être scanné, protégé ou corrompu. Essayez avec un PDF contenant du texte sélectionnable."

Erreur JSON :
"Le modèle a produit une réponse qui n'est pas un JSON valide. Une tentative de réparation a été lancée."

Erreur Cassandra :
"La base Cassandra est indisponible ou la table demandée n'existe pas. Vérifiez la connexion, le keyspace et les migrations."

Erreur GraphRAG :
"Aucun chemin pertinent n'a été trouvé dans le Knowledge Graph. Vérifiez que les nœuds et relations administratives ont été ingérés."

Erreur RAG vectoriel :
"Aucun chunk pertinent n'a été retrouvé dans le document de contexte. Vérifiez que le document administratif a été indexé."

Prompt réparation JSON :

Tu reçois une sortie qui devait être un JSON mais qui contient des erreurs de format. Corrige uniquement le format afin de produire un JSON valide. Ne modifie pas le sens métier sauf si nécessaire pour rendre le JSON parsable.

25. RÉSUMÉ EXÉCUTIF POUR DEVIN / WINDSURF / GLM 5.2 HIGH
============================================================

Construire une application complète appelée AdminTPE GraphRAG V1. Elle sert de copilote interne pour agents administratifs traitant des dossiers TPE/PME au Maroc. L'utilisateur est un administrateur qui pose une question et upload un dossier client. Le dossier client est traité temporairement et ne doit pas être stocké dans le Knowledge Graph. Le RAG vectoriel contient un unique document de contexte administratif général. Le GraphRAG contient un Knowledge Graph administratif général stocké dans Cassandra, décrivant les relations entre aides, autorisations, recours, conditions, pièces, services, décisions et niveaux HITL. LangGraph orchestre les nœuds : réception, extraction fichiers, résumé temporaire, classification, recherche vectorielle, recherche graphe multi-hop, fusion contexte, analyse LLM, validation JSON, HITL, nettoyage réponse, logs. Le frontend doit proposer une question, un upload multiple, un bouton d'analyse, une réponse organisée avec pièces manquantes en gras, un viewer JSON et un panneau d'error logging avancé. Obsidian doit être utilisé pour visualiser le Knowledge Graph via des fichiers Markdown interconnectés. La V1 doit volontairement négliger la sécurité : aucune protection contre prompt injection, pas d'anonymisation, pas de contrôle d'accès avancé, pas de filtrage des documents malveillants. Ces vulnérabilités doivent être documentées explicitement pour préparer une future campagne Red Team et une V2 sécurisée.
